# SPF, DKIM, etc.

## Deliverability

If you are using the DMS container to deliver mail to a domain like Gmail, you need to ensure that your domain is properly configured with SPF, DKIM, and DMARC records. You can do this by setting up your domain with the appropriate records, or by using a relay service if you don't have a static public IP or prefer not to set up your records.

The below explains how to set up your domain with SPF, DKIM, and DMARC records. It was generated by Opus 4.7. If you have questions about how to best your agents' email, try using your coding agent from within this repository.

*The below was Generated by Opus 4.7*

Here's the playbook, using your actual domain (e.g., agents.acme.com, mail host mail.agents.acme.com → replace with your own) so you can copy/paste. The order matters — Gmail's bar in 2026 is essentially "all four of PTR, SPF, DKIM, DMARC pass and align," so skipping any one of them will still get you 5.7.26 (failed delivery).

## 0\. Prerequisites you have to have
* A **static public IP** that you own (not a residential/dynamic IP, not a NAT'd home IP). Most ISPs block port 25 outbound from consumer lines anyway, and Gmail/MS scoreboards heavily penalise dynamic ranges. If you're running this on a homelab and only have a residential IP, stop here and use a relay instead. You can't fight this one with DNS.
* Control of the agents.acme.com DNS zone (you need to add TXT and DNS records).
* The DMS container reachable from the public internet on port 25 inbound (for bounces / replies coming back) and able to make outbound port 25.

## 1. PTR (reverse DNS) — set with your hosting/IP provider
PTR is the one record you **cannot set yourself in DNS**; whoever assigned you the IP (your VPS host, ISP, cloud provider) controls the reverse zone. Open a ticket / use their console:

> Please set PTR for <your.public.ip> to mail.agents.acme.com. (note the trailing dot).

Also confirm forward-confirmed reverse DNS (FCrDNS): mail.agents.acme.com must resolve forward to that same IP. So add the forward A record too:

```
mail.agents.acme.com.   IN  A   <your.public.ip>
```

Verify:

```
dig +short mail.agents.acme.com            # → your IP
dig +short -x <your.public.ip>             # → mail.agents.acme.com.
```

Both must match. If they don't, **don't proceed** — fix this first, otherwise nothing downstream will help.

While you're in the DNS zone, also make sure mail for the domain points at this host:
```
agents.acme.com.        IN  MX  10  mail.agents.acme.com.
```

## 2\. SPF — TXT record at the apex
Tell the world which IPs may send From: …@agents.acme.com. The minimal record when DMS is your only sender:
```
agents.acme.com.   IN  TXT  "v=spf1 mx -all"
```

mx resolves to the IP behind your MX, which is the DMS host — same machine doing the sending, so it passes. -all is "hard fail anything else." If you also send through Google Workspace or a relay, add it: "v=spf1 mx include:_spf.google.com -all".

One SPF record per domain. Don't add a second.

Verify (after DNS has propagated, give it 5–10 min):
```
dig +short TXT agents.acme.com
```

## 3\. DKIM — generate key in DMS, publish TXT
DMS has a built-in helper. From the host running the DMS container:
```
docker exec -ti dms setup config dkim domain agents.acme.com
```
This generates a 2048-bit key under services/dms/config/opendkim/keys/agents.acme.com/mail.txt (the selector is mail by default) and wires opendkim into Postfix. 

Open that file:
```
cat services/dms/config/opendkim/keys/agents.acme.com/mail.txt
```

It'll look like:
```
mail._domainkey   IN   TXT   ( "v=DKIM1; h=sha256; k=rsa; "
    "p=MIIBIjANBgkqhki…long…AB" )
```

Take everything inside the quotes (concatenated, no quotes, no parentheses) and publish it in your DNS (a TXT record):
```
mail._domainkey.agents.acme.com.   IN  TXT  "v=DKIM1; h=sha256; k=rsa; p=MIIBIjANBgkqhki…AB"
```

If your DNS provider has a character limit on TXT values (255 is the per-string limit, but most UIs accept the whole thing and split for you), paste the whole string. If they reject it as too long, split into multiple quoted segments — DNS concatenates them.

Restart DMS so opendkim picks up the key:
```
docker compose restart dms
```

Verify:
```
dig +short TXT mail._domainkey.agents.acme.com
```
should return your p=… blob.

## 4\. DMARC — TXT record at_dmarc
This is the policy that ties SPF and DKIM together and tells receivers what to do on failure. Start in **monitoring mode** so you can see what's actually happening before you tighten:
```
_dmarc.agents.acme.com.   IN  TXT  "v=DMARC1; p=none; rua=mailto:postmaster@agents.acme.com; fo=1; adkim=r; aspf=r"
```

* p=none = "don't reject, just send me reports." Gmail/Yahoo aggregate reports will arrive at rua= within a day or two. Read them. Once you see SPF and DKIM passing consistently, tighten to p=quarantine and then p=reject. Jumping straight to p=reject is how legitimate mail ends up in the spam folder for two weeks while you debug.
* adkim=r/aspf=r = relaxed alignment (subdomain mail.agents.acme.com aligns with agents.acme.com). Strict would require an exact domain match.

## 5. Smoke-test the whole stack
Send a test message **from your agent to** check-auth@verifier.port25.com. (To send as your agent, you can use an email client configured with your agents' accounts see this [README](config/README.md)) It replies with a full audit covering PTR, HELO, SPF, DKIM signature validity, DKIM alignment, DMARC, and a SpamAssassin score. Read that reply carefully — it will name exactly which check is failing.
Alternatively, send to a Gmail account you control and use **"Show original"**: look for the Authentication-Results header. You want to see:
```
Authentication-Results: mx.google.com;
dkim=pass header.i=@agents.acme.com …
spf=pass (google.com: domain of chris@agents.acme.com designates <ip> as permitted sender) …
dmarc=pass (p=NONE …) header.from=agents.acme.com
```
All three pass. If any are neutral, softfail, or fail, that's your next thing to fix.

## 6\. Things that quietly bite you
* **HELO name mismatch.** Postfix's HELO must match the PTR. In DMS, set OVERRIDE_HOSTNAME if your container hostname differs from your public FQDN, or just make sure the compose hostname: matches the FQDN with the PTR.
* **DKIM signing the wrong domain.** Run docker exec dms setup config dkim with --help to see selectors and domain flags. If you send From: @agents.agents.acme.com, you need a DKIM key for *that* domain too — the From-domain has to be signed for alignment.
* **TLS cert.** Self-signed is fine for opportunistic TLS (Gmail accepts it for outbound delivery in your direction, as your earlier log showed). It is **not** fine for MTA-STS. You don't need MTA-STS for Gmail to accept you, but if you eventually want it, you'll need a Let's Encrypt cert covering mail.agents.acme.com.
* **Reputation warmup.** Even with all four records green, a brand-new IP sending its first message to Gmail can land in spam. Send slowly for the first week or two; don't blast.
* **Port 25 blocked outbound.** Many cloud providers (AWS, GCP, Azure, OVH on default plans, most home ISPs) block outbound 25 by default. Test: nc -vz gmail-smtp-in.l.google.com 25 from the DMS host. If it hangs, file a request with your provider to unblock — or, again, use a relay.

If after all of this you still get 5.7.26, the verifier@port25 audit will tell you exactly which signal failed, and the answer is almost always either PTR or DKIM alignment. Start there.
