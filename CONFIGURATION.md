# CONFIGURATION.md — LLM-Guided Kikubot Setup

This document tells **you, the LLM agent**, how to walk a user through configuring a Kikubot deployment as a multi-step chat. Read this whole document once, then drive the conversation step-by-step. Do not dump the full plan on the user up front — work through one step at a time, asking only the questions that step requires, and confirming or validating before moving on.

## How to use this document

- You are the configuration assistant. The user is a human operator setting up Kikubot on their machine.
- Treat the **working directory** as the Kikubot repository root (the directory containing `docker-compose-example.yml`, `configs/`, `services/`, `scripts/`).
- After every change, briefly summarise what you wrote and where, so the user can audit.
- Never invent values. If you don't know something, ask. If a value is sensitive (API keys, passwords), do not echo it back or store it anywhere except the file the user expects.
- All shell commands you suggest should be run from the repository root unless explicitly noted.
- If the user has already partially configured the deployment, **read the existing files first** and offer the existing values as defaults the user can keep or change.
- At the end of every section, ask the user to confirm before moving on.

A Kikubot deployment has exactly two configuration files plus an optional bundled mail server:

| File | Purpose | Tracked in git? |
|---|---|---|
| `configs/agents.yaml` | Non-secret roster + common defaults + per-agent overrides | No (a `configs/agents-example.yaml` template ships with the repo) |
| `configs/secrets.env` | LLM API keys, per-agent mailbox passwords, tool credentials | No (a `configs/secrets-example.env` template ships with the repo) |
| `docker-compose.yml` | One service per agent, regenerated by the configurator | No (`docker-compose-example.yml` ships with the repo) |

---

## Step 1 — Requirements

Goal: confirm the user has Docker Compose (or a compatible runtime).

1. Run a quick check:
   ```bash
   docker compose version
   ```
   If that succeeds, you're done with this step. If it fails, try:
   ```bash
   docker-compose version
   podman compose version
   ```
2. If none of those exist, **stop** and tell the user that Kikubot requires Docker Compose v2 (or a compatible drop-in like `podman compose`) and point them at https://docs.docker.com/compose/install/. Do not proceed until they confirm an install.
3. If a compatible runtime is present, tell the user which one was detected and ask them to confirm it's the one they want to use.

---

## Step 2 — `configs/agents.yaml`

This is the single source of truth for non-secret deployment config. It has two top-level sections:

```yaml
common:
  email_server: mail.agents.example.com:993
  smtp_server: mail.agents.example.com:587
  email_insecure_tls: false
  max_history_chars: 200000
  max_tokens: 6000
  agent_timeout: 300
  max_turns: 20
  system_prompt: |
    You are a helpful agent...
    {{coworkers}}
  coordinator_system_prompt: |
    You are a helpful Coordenator Agent...
    {{coworkers}}

agents:
  - name: Kiku
    email: kiku@agents.example.com
    role: Coordinator
    description: Communicates with users. Coordinates other agents.
    tools: [report, snooze]
    # Any common: field may be overridden here. Missing fields inherit from common:.
    llm_provider: openrouter
    llm_model: anthropic/claude-sonnet-4.6
    llm_openrouter_backup: [z-ai/glm-5.1]
    max_history_chars: 300000
    max_turns: 40
    whitelist: [example.com, agents.example.com]
    system_prompt: |
      Kiku, you are a helpful Coordenator Agent...
      {{coworkers}}
```

### 2a. Seed `configs/agents.yaml`

1. If `configs/agents.yaml` does not exist, copy the example:
   ```bash
   cp configs/agents-example.yaml configs/agents.yaml
   ```
2. Walk through the `common:` block field by field. For each value, show the current setting (from `configs/agents.yaml`, falling back to `configs/agents-example.yaml`) and ask the user to keep or change.

   | Field | Prompt | Notes |
   |---|---|---|
   | `email_server` | "IMAP server (`host:port`) for the agents' inboxes." | e.g. `mail.agents.example.com:993`. |
   | `smtp_server` | "SMTP server (`host:port`) for outgoing mail." | Defaults to port 587 if no port is given. |
   | `email_insecure_tls` | "Accept self-signed TLS certs from the mail server?" | `true`/`false`. Required `true` if Step 4 uses self-signed certs. |
   | `max_history_chars` | "Max characters of per-thread history retained per turn." | Default `200000`. |
   | `max_tokens` | "Max LLM output tokens per response." | Default `6000`. |
   | `agent_timeout` | "Per-message processing deadline, in seconds." | Default `300`. |
   | `max_turns` | "Per-message turn budget." | Default `20`. Coordinators usually want `40`. |
   | `system_prompt` | "Default system prompt." | May include `{{coworkers}}`. Multiline literal block (`|`). |
   | `coordinator_system_prompt` | "Recommended seed for coordinator agents." | Used by the configurator's `Paste Coordinator prompt` button. |

### 2b. Per-agent entries

Ask the user how many agents they want, then for each entry under `agents:` collect:

| Field | Prompt | Notes |
|---|---|---|
| `name` | "Display name (e.g. `Kiku`)." | Required. |
| `email` | "Full email address." | Required. The local-part (lowercased) becomes the agent's stem — used as the data volume name and the password key in `secrets.env`. |
| `role` | "Short tag-like label (e.g. `Coordinator`, `Social Media`, `CRM`)." | Required — surfaces in peer agents' system prompts. |
| `description` | "One-sentence summary of what the agent does." | Required — surfaces in peer agents' system prompts. |
| `tools` | "List of registry keys (see Step 3)." | Required, may be `[]`. |
| Override knobs | "Should this agent override any common: value?" | Only ask if the user wants per-agent customisation. Any `common:` field is overridable. |
| `whitelist` / `blacklist` | "Allowed / denied senders, list of emails or domains." | Mutually exclusive. |
| `llm_provider` | "Provider: `anthropic` or `openrouter`." | Must have its API key set in `secrets.env`. |
| `llm_model` | "Model id (provider-specific tag)." | e.g. `claude-sonnet-4-6` or `anthropic/claude-sonnet-4.6`. |
| `llm_openrouter_backup` | "Ordered list of fallback model ids." | YAML list. Only ask if provider is `openrouter`. |
| `system_prompt` | "Override the default prompt for this agent." | Optional. May include `{{coworkers}}`. If the agent is a coordinator, offer to seed from `coordinator_system_prompt`. |
| `max_history_chars` / `max_turns` | "Override conversation budgets?" | Optional. Coordinators typically need a higher `max_turns` (e.g. `40`). |

**Validation per agent:**
- `whitelist` and `blacklist` must not both be set.
- `llm_provider` (effective, after override) must have its key in `secrets.env`.
- `email` must contain `@`.

## Step 3 — `configs/secrets.env`

Secrets never go in `agents.yaml`. Every container loads `configs/secrets.env` via `env_file`. Convention:

- `ANTHROPIC_API_KEY` / `OPENROUTER_API_KEY` — at least one is required.
- `<UPPERCASED_LOCAL_PART>_EMAIL_PASSWORD` — one per agent. The container resolves its own by uppercasing the local-part of `AGENT_EMAIL`. Example: agent `kiku@…` reads `KIKU_EMAIL_PASSWORD`.
- Tool credentials (`SALESFORCE_CLIENT_ID`, `BUFFER_API_KEY`, etc.) — only required for agents that actually use the tool.

1. If `configs/secrets.env` does not exist, copy the template:
   ```bash
   cp configs/secrets-example.env configs/secrets.env
   ```
2. **Do not collect secret values from the user in chat.** Tell them:
   > "Open `configs/secrets.env` and fill in the values for the LLM provider(s) you'll use, one `*_EMAIL_PASSWORD` per agent, and the credentials for any tools you assigned in Step 2."
3. You may read the file to confirm a placeholder has been replaced, but do **not** display secret values back.

## Step 4 — Tool registry

Run the tools_list script to enumerate every tool the running binary supports:

```bash
go run ./scripts/tools_list
```

This prints a JSON array. Each entry has:
- `key` — the value to put in `agents.yaml` under `tools:` (empty `key` means the tool is a core tool, always loaded — do **not** list core tools).
- `name` — the tool name the LLM sees at runtime.
- `description` — what the tool does.
- `core: true` — present only on core tools.

### Tool selection guidance

- **Coordinator agents should include `snooze`** so they can schedule recurring or one-off tasks.
- **`report`** is recommended for coordinator agents so they reply to users in a structured form.
- Some tools require credentials in `secrets.env`. Before assigning, confirm the credentials are set:
  - `salesforce_mcp` → `SALESFORCE_CLIENT_ID`, `SALESFORCE_CLIENT_SECRET`, `SALESFORCE_INSTANCE_URL`
  - `buffer_mcp` → `BUFFER_API_KEY`
  - `xero_mcp` → `XERO_CLIENT_ID`, `XERO_CLIENT_SECRET`
  - `tavily_mcp` → `TAVILY_API_KEY`
  - `helpjuice` → `HELPJUICE_API_KEY`, `HELPJUICE_ACCOUNT`
  - `wordpress` → `WEBSITE_URL`, `WORDPRESS_USER`, `WORDPRESS_PASSWORD`
  - `mxmcp` → `MXMCP_API_KEY`
  - `box_cli` → a Box JWT app config dropped at `box_config.json`
  - `file_text` → `TIKA_URL` (defaults to the bundled Tika sidecar)
  - `anthropic_web_search` → only works when `llm_provider: anthropic`. Warn the user if they try to assign it to an openrouter agent.
- Tool order is preserved as-listed.

## Step 5 — Email service

Kikubot agents communicate by email, so the user must have an IMAP+SMTP server available. Two paths:

### Path A — User-provided mail server

If the user already has a mail server (Gmail, Office 365, FastMail, their own postfix, etc.):
- Confirm `common.email_server` / `common.smtp_server` are correct.
- Remind them that each agent needs a real mailbox on that server, with the matching `<STEM>_EMAIL_PASSWORD` in `secrets.env`.

### Path B — Bundled docker-mailserver sidecar (optional)

Ask: **"Do you want to use the project's bundled email server (docker-mailserver at `services/dms/`)?"**

If **yes**:

1. **Localhost question.** Ask: "Do you want the mail server to be reachable as `localhost` from the agent containers?" If yes, the user must map the chosen hostname to localhost inside each container via `extra_hosts:` in `docker-compose.yml` (uncomment the relevant line, replace the hostname).
2. **Compose file.**
   ```bash
   cp services/dms/docker-compose-example.yml services/dms/docker-compose.yml
   ```
3. **Edit `services/dms/docker-compose.yml`**:
   - `hostname:` → mail server's public hostname (e.g. `mail.agents.example.com`)
   - `domainname:` → parent domain (e.g. `agents.example.com`) — must match the `@<domain>` of every agent email
4. **Postfix policy files** (optional). The configurator's email-service page generates `services/dms/config/postfix-transport.cf` and `postfix-sender-access.cf` to limit which external domains agents may send to / receive from. Format is documented in `services/dms/config/`.
5. **TLS certificates.** The mail server needs `fullchain.pem` and `privkey.pem` at `services/dms/certs/`.
   - For production, supply real certs.
   - **For dev/test, offer to generate self-signed certs:**
     ```bash
     mkdir -p services/dms/certs
     openssl req -x509 -newkey rsa:4096 -sha256 -days 3650 -nodes \
       -keyout services/dms/certs/privkey.pem \
       -out   services/dms/certs/fullchain.pem \
       -subj  "/CN=<hostname>" \
       -addext "subjectAltName=DNS:<hostname>,DNS:<domainname>"
     ```
     After generating, set `email_insecure_tls: true` in `configs/agents.yaml`.
6. **Bring it up:**
   ```bash
   cd services/dms && docker compose up -d
   docker compose ps
   ```
7. **Add mailboxes** — one per agent, password matching `<STEM>_EMAIL_PASSWORD`:
   ```bash
   docker exec -it dms setup email add <agent>@<domain> "<password>"
   docker exec -it dms setup email list
   ```

> "If you'd like to inspect or further modify the configuration in a web dashboard, you can run `go run ./scripts/configurator` (serves on `127.0.0.1:50042` by default). It edits the same `configs/agents.yaml` + `configs/secrets.env` we just touched, and regenerates `docker-compose.yml` whenever you add or edit an agent."

This is informational only; do not insist they use it.

## Step 7 — `docker-compose.yml`

The configurator regenerates this file automatically. For a hand-run flow:

1. ```bash
   cp docker-compose-example.yml docker-compose.yml
   ```
2. For each agent under `agents:` in `configs/agents.yaml`, ensure a corresponding service exists:
   - Service name: lowercase, non-alphanumeric runs collapsed to `-` (e.g. `Kiku Coordinator` → `kiku-coordinator`).
   - `env_file: configs/secrets.env`
   - `environment:` includes `RUNNING_IN_CONTAINER=true` and `AGENT_EMAIL=<the agent's email>`. **AGENT_EMAIL is the identity selector** — the container uses it to find its entry in `agents.yaml`.
   - `volumes:` mounts `./data/<stem>:/app/data` where `<stem>` is the lowercased local-part of the email.
3. **If `email_server` points to `localhost`** (Step 5, Path B with the localhost option), uncomment the optional `extra_hosts:` line in each service:
   ```yaml
   extra_hosts:
     - "host.docker.internal:host-gateway"
     - "mail.agents.example.com:host-gateway"   # ← uncomment, replace hostname
   ```

## Step 8 — Validate

Before launching, run the doctor:

```bash
go run scripts/kikudoctor/*.go
```

- Clean report → ready to launch with `docker compose up -d --build`.
- Findings → walk the user through each one. Common categories:
  - Missing or malformed entries in `configs/agents.yaml`.
  - `configs/secrets.env` missing an LLM key or a `<STEM>_EMAIL_PASSWORD`.
  - Tool keys assigned to agents whose required credentials are blank.
  - `docker-compose.yml` services not aligned with the roster.

Iterate until clean.

---

## Conversation pacing

- One topic per turn. Do not ask three questions when the next decision depends only on one.
- Read existing files before asking — if a value is already on disk, present it as a default and ask "keep, or change?"
- When you write a file, show a diff (or the full final contents if it's small) and ask for confirmation.
- Never paste secrets back in chat. Refer to them as "the value you put in `secrets.env`" instead.
- After Step 8 passes, the deployment is configured. Hand off with the launch command, the log-tailing command (`docker compose logs -f`), and a reminder that they can re-run `scripts/configurator` or `scripts/kikudoctor` at any time.
